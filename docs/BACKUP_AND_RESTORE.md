# BACKUP_AND_RESTORE.md

## Цель
Обеспечить легкий откат и восстановление Coach:
- по коду;
- по конфигам;
- по данным;
- после неудачного релиза;
- после падения сервера.

## Политика
1. Каждый логически завершенный шаг фиксировать commit'ом.
2. Каждый commit кратко описывать в `docs/CHANGELOG.md`: что сделано и зачем.
3. Перед опасными изменениями структуры/данных делать snapshot.
4. Хранить код отдельно от runtime-данных.
5. Регулярно выносить резервные копии данных за пределы сервера.

## Структура
- `app/` — код приложения
- `infra/` — docker/deploy
- `storage/` — локальное файловое хранилище runtime
- `backups/` — локальные snapshots проекта и метаданных
- `docs/CHANGELOG.md` — журнал изменений
- `docs/STATE.md` — текущее состояние проекта

## Минимальный backup-контур v1
- Код: git + GitHub repo `coach`
- Данные: архив snapshots в `backups/`
- Документы пользователя: отдельно архивировать и выносить из сервера
- База данных: позже добавить pg_dump snapshots

## Правило перед risky-изменениями
Перед миграциями схемы, крупных рефакторингов ingestion и структуры данных:
1. сохранить git commit
2. сделать tar.gz snapshot проекта
3. отдельно сохранить export данных
4. только потом менять код

## Базовый сценарий восстановления
1. Клонировать repo `coach`
2. Развернуть `infra/`
3. Восстановить snapshot данных
4. Поднять контейнеры
5. Проверить `/health`
